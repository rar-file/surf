## Review Feedback

The following issues were found during code review. Fix these specific issues. Do NOT re-do the entire migration — only address the problems below.

### Issue 1 [MAJOR]: core/ai_search.py

**Problem:** Silent fallback to DuckDuckGo when `provider='tavily'` is explicitly requested but `TAVILY_API_KEY` is not set. In `AISearch.__init__`, `_resolve_search_provider('tavily')` returns `'tavily'`, but the subsequent `if api_key:` guard leaves `self._tavily = None` when the env var is absent, causing all searches to silently use DuckDuckGo. The same silent fallback exists in the module-level `news_search()` function (lines 387–392). A user who passes `--search-provider=tavily` without an API key gets DuckDuckGo results with no indication their selection was ignored.
**Suggestion:** When `provider` is explicitly `'tavily'` and `TAVILY_API_KEY` is absent, raise a clear `ValueError` (or at minimum print a prominent warning) rather than silently falling back. For example in `AISearch.__init__`: `if self._provider_name == 'tavily' and not api_key: raise ValueError("TAVILY_API_KEY environment variable is required when provider='tavily'")`. Apply the same guard in the module-level `news_search()` function.

### Issue 2 [MINOR]: .migration-plan-surf-duckduckgo-add-tavily.md

**Problem:** A migration planning artifact file was committed to the repository root. This file is implementation scaffolding that has no value in the final codebase and should not be part of the source history.
**Suggestion:** Remove `.migration-plan-surf-duckduckgo-add-tavily.md` from the repository. Add a `.gitignore` entry for `.migration-plan-*.md` if other such files might appear in future migrations.

### Issue 3 [MINOR]: core/ai_search.py

**Problem:** In `TavilyBackend.news_search()` (line 81), the `source` field is hardcoded to an empty string `"source": ""`, discarding source attribution that Tavily does return in its response payload.
**Suggestion:** Change `"source": ""` to `"source": r.get("source", "")` to capture the source domain/name that Tavily provides, keeping the result schema consistent with what the DuckDuckGo path returns.

### Issue 4 [MINOR]: core/ai_search.py

**Problem:** The module-level `news_search()` function (lines 375–420) bypasses `AISearch` and directly instantiates `TavilyBackend`, duplicating the provider-resolution logic already in the class. The class-based path uses `self._tavily`, while this function re-resolves and re-creates the backend inline, leading to two separate code paths that must be kept in sync.
**Suggestion:** Refactor `news_search()` to delegate through `AISearch`, similar to how `search()` does: instantiate `AISearch(provider=provider)`, then call a `news_search()` method on it. Move the DuckDuckGo news DDGS logic into `AISearch` as a method, keeping the logic in one place.
